### 首次审查发现的严重问题修复情况

#### ✅ 1. Cross-Encoder 重排实现 - 已修复
- **修复内容**：
  - 新增 `backend/sbo_core/rerank_client.py`，实现了 `RerankClient` 类
  - 支持通过 HTTP 调用外部 rerank provider
  - 实现了完整的错误处理（超时、连接失败、认证失败、服务器错误）
  - 返回结构化的 `RerankResult` 对象
  - 在 `retrieval_pipeline.py` 中集成了真实的 rerank 调用
  - 实现了降级策略：rerank 失败时回退到 fusion 结果
- **配置支持**：
  - 在 `config.py` 中添加了 rerank 相关配置项：
    - `rerank_provider_url`：rerank 服务地址
    - `rerank_api_key`：API 密钥
    - `rerank_model_id`：模型 ID
    - `rerank_timeout_ms`：超时时间（50-60000ms）
    - `rerank_weight`：重排权重（0-1）
    - `rerank_max_candidates`：最大候选数（1-100）
  - 配置项有完整的校验逻辑
- **测试覆盖**：
  - `test_retrieval_pipeline.py` 中新增了 rerank 相关测试：
    - `test_optional_rerank_success`：测试重排成功场景
    - `test_optional_rerank_degradation`：测试重排降级场景
  - 使用 `httpx.MockTransport` 模拟 HTTP 调用
  - 验证了重排分数的加权融合逻辑
  - 验证了降级服务记录机制
- **审计日志**：
  - 成功时记录 provider_url、model、timeout_ms、weight、candidates_in、results_out
  - 降级时记录错误信息
- **评价**：实现完整，符合设计要求，降级策略清晰

#### ✅ 2. 生命周期访问计数更新机制 - 已修复
- **修复内容**：
  - 新增 `backend/sbo_core/services.py` 中的 `EvidenceAccessService` 类
  - 实现了 `record_access` 方法，支持批量更新访问计数
  - 在 `query_service.py` 中集成了异步访问计数记录
  - 使用 `asyncio.create_task` 异步更新，不阻塞查询响应
  - 更新逻辑：
    - 如果记录不存在，创建新记录（access_count=1）
    - 如果记录存在，增加 access_count 并更新 last_accessed_at
- **数据库支持**：
  - 使用 `EvidenceAccessStats` 表存储访问统计
  - 字段包括：user_id、evidence_id、access_count、last_accessed_at
- **错误处理**：
  - 访问计数更新失败不影响查询响应
  - 失败时记录日志，返回更新数量 0
- **测试覆盖**：
  - 虽然 `test_retrieval_pipeline.py` 未直接测试访问计数，但 `query_service.py` 中的集成逻辑清晰
  - 建议：后续可在 `test_query.py` 中增加访问计数更新的集成测试
- **评价**：实现完整，符合设计要求，异步更新不阻塞主流程

### 🟡 一般问题（5）

#### 1. 硬编码的默认用户 ID
- **位置**：`backend/sbo_core/routes/ingest.py:chat_interaction`、`backend/sbo_core/routes/manage.py`
- **原因**：使用硬编码的 `user_id = "default_user"` 或 `_DEFAULT_USER_ID`
- **影响**：单机单用户形态可接受，但未来扩展为多用户时需要重构
- **建议**：添加 TODO 注释说明未来需要从认证信息中获取真实用户 ID
- **状态**：未修复（可接受，属于阶段 1 简化）

#### 2. LLM 调用未实现
- **位置**：`backend/sbo_core/routes/ingest.py:chat_interaction`
- **原因**：LLM 生成回复部分仅返回占位符文本
- **影响**：`/chat` 接口无法提供真实的对话能力
- **建议**：实现 LLM 调用逻辑（对接 llama.cpp 或外部 provider）
- **状态**：未修复（可接受，属于阶段 1 简化）

#### 3. 文件存储未实现
- **位置**：`backend/sbo_core/routes/ingest.py:upload_file`
- **原因**：文件上传后未实际保存到存储系统
- **影响**：文件上传功能不完整，无法实际使用
- **建议**：实现文件存储逻辑（本地文件系统或对象存储）
- **状态**：未修复（可接受，属于阶段 1 简化）

#### 4. 检索答案提示生成过于简单
- **位置**：`backend/sbo_core/query_service.py:_generate_answer_hint`
- **原因**：答案提示生成仅拼接前两条证据文本，未使用 LLM
- **影响**：答案提示质量较低，用户体验不佳
- **建议**：实现基于 LLM 的答案提示生成
- **状态**：未修复（可接受，属于阶段 1 简化）

#### 5. 语义相似度计算过于简化
- **位置**：`backend/sbo_core/retrieval_pipeline.py:_calculate_semantic_similarity`
- **原因**：使用 Jaccard 相似度 + 随机性模拟向量相似度
- **影响**：检索质量无法达到生产级别
- **建议**：实现真实的向量相似度计算（使用 embeddings）
- **状态**：未修复（可接受，属于阶段 1 简化）
