# Project Rules (AGENTS) - SecondBrainOS

## 0) 绝对红线 (Absolute Directives)
所有 AI 助手在执行任务时，必须将以下原则置于最高优先级：
1. **中文沟通**：始终使用中文回复与编写文档（代码注释、英文错误码除外）。
2. **防过度设计 (KISS & YAGNI)**：
   - **严格禁止**超前实现、预测性编码、炫技式抽象（如无谓的接口/基类/泛型体操）。
   - **只实现当前任务（T#）明确要求的功能**。多写一行无用代码视为 Blocker。
   - 首选 Python 标准库 / TS 原生方法，拒绝重复造轮子。
3. **真实集成 (No Fake Mocks)**：
   - 必须连接真实服务（Postgres, Redis, Neo4j, LLM API）进行集成与冒烟测试。
   - **严禁**使用 mock 对象或 `test.skip()` 跳过测试。缺少配置导致失败必须抛错并要求用户补充。
4. **零技术债务 (Zero Tech Debt)**：
   - 任务完成时不得留下任何 `TODO`、`待优化`。所有审查报告中的问题（Blocker/Critical/Major）必须 100% 修复。

## 1) 编码与架构规范

### 1.1 全栈语言与校验边界
- **前端 (TS)**：强制 TypeScript Strict（**禁止 `any`**）。输入/输出必须使用 **Zod** 校验并推导类型。
- **后端 (Python)**：强制类型注解。所有对外出入参必须使用 **Pydantic** 模型作为单一事实源。
- **错误处理**：抛出的异常必须结构化，`message` 必须使用**英文**，便于日志搜索。

### 1.2 配置外部化 (强制)
- **禁止硬编码**：URL、端口、Token、魔法数字必须外部化。
- **配置加载与校验**：必须通过环境变量或 `.env.local` 读取，加载时强制使用 Zod/Pydantic 校验类型与必填项。
- **文档同步**：新增配置必须同步到 `.env.example`。

### 1.3 SecondBrainOS 专属架构
- **核心层**：FastAPI + Redis Queue (RQ) 处理异步巩固任务。
- **存储层**：Postgres (主存/pgvector) + Neo4j (知识图谱) + Redis (缓存/队列)。
- **核心 API 契约**：
  - `POST /ingest`：写入原始事件 + 异步入队（快确认）。
  - `POST /query`：基于 vector / graph 召回证据（`mode=fast/deep`）。
  - `POST /forget`：异步擦除。

## 2) 文档生成与任务状态

### 2.1 文档落盘规则
- 必须以 **Markdown (.md)** 格式在实际目录中创建/更新文件，**绝不能只在聊天窗口输出**。
- 若写入失败（如被中断），必须在聊天中用 ` ```markdown ` 代码块输出完整内容供用户复制。

### 2.2 任务提示词联动 (强制)
- 当更新非日志类文档（如 API 规范、设计方案）时，**必须同步更新后续未开始任务的 Prompt**（`docs/task-prompts/*.md`）：
  - 在 Prompt 中追加“权威参考文档”引用。
  - 勾选该任务 Prompt Checklist 中的“Doc References Updated”。

### 2.3 任务完成标记
- 只有在**测试全部通过且生成验收日志后**，才能在 `docs/tasks.md` 中打钩 `[x]` 并标记完成。

## 3) 自动化验收与测试规范

### 3.1 测试双覆盖要求
每个任务必须同时具备以下两类测试，并在验收日志中记录执行结果：
1. **单元测试 (Unit Test)**：覆盖核心逻辑、边界条件和结构化错误路径。允许使用依赖注入模拟服务。
2. **冒烟测试 (Smoke Test)**：端到端真实链路验证。必须涵盖真实的 DB 读写或真实的外部 API 调用。

### 3.2 验证命令执行标准
- **后端 (Python)**：
  - 必须使用项目根目录的虚拟环境（如 `.venv/bin/python`）。
  - 最低编译校验：`.venv/bin/python -m compileall backend`
  - 单测执行：`.venv/bin/python -m pytest -q backend/tests`
- **前端 (Web)**：执行 `npm run build` 确保无类型或打包错误。
- *注：若因环境缺失无法运行，必须说明原因并制定补测计划，不得直接标记完成。*

## 4) 审查与验收交付物

### 4.1 报告命名规范
- **验收日志**：`reports/YYYY-MM-DD_T{N}_short-slug.md`
- **审查报告**：`reports/YYYY-MM-DD_T{N}_short-slug-review.md`

### 4.2 审查阻断红线 (Review Blockers)
审查 AI 发现以下问题将直接阻断验收，开发 AI 必须无条件返工：
| 违规类型 | 表现特征 | 强制动作 |
|---------|---------|---------|
| **过度设计** | 包含当前需求外未使用的字段、API、多余的工厂模式/基类 | 立即删除冗余代码 |
| **契约漂移** | 接口实现与 `docs/contracts/` 定义的 Schema 不一致 | 对齐权威设计文档 |
| **未校验配置** | `os.getenv` 后直接使用，没有异常捕获和类型转化 | 引入 Pydantic/Zod 校验 |
| **死代码** | 存在未使用的变量、私有函数、被大段注释掉的历史代码 | 立即清理 |
| **测试造假** | 验收日志说测试通过，但使用了 `mock` 替代真实数据库 | 改为真实服务冒烟测试 |

### 4.3 零遗留项声明
验收日志末尾必须包含明确的**【零遗留项声明】**：
- [ ] 所有审查问题均已修复。
- [ ] 没有任何 TODO 或未决项。
- [ ] 新增 API/配置已完整补充至对应文档。
